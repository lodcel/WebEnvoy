# Execution Context

本文件定义 Loom 当前最小执行上下文合同。

本文件当前承接：

- `EXT-0011`
- `EXT-0038` 的每轮读取部分

完整时序见 [execution-chain.md](./execution-chain.md)。

## 1. 能力定位

执行上下文用于把单次执行轮次绑定到同一组稳定事实。

它服务：

- 开始执行前的读取
- `checkpoint` 与 `resume` 的对齐
- `handoff`、review 与放行前的事实对齐

它不负责保存恢复内容本身；恢复内容由 [recovery-model.md](./recovery-model.md) 承接。
字段归属见 [fact-chain-contract.md](./fact-chain-contract.md)。

## 2. 最小语义集合

每次正式执行前，至少要能确定以下语义：

- 当前事项是谁
- 当前事项属于哪条执行路径
- 当前目标是什么
- 当前范围边界是什么
- 当前工作现场是什么
- 当前恢复主入口是什么
- 当前 checkpoint 阶段是什么
- 当前验证入口是什么
- 当前环境 lane 是什么

Loom 固化的是语义集合，不固化字段名、编号格式或平台来源。
这些语义本身属于派生读取，不是独立 authored 真相。

## 3. 每轮读取最小事实集

进入正式执行轮次前，默认先读取以下事实：

- 当前 `Work Item` 或等价执行入口
- 当前恢复主入口中的最近停点、下一步与阻断项
- 当前 checkpoint 阶段与执行路径
- 当前工作现场映射与最近 git 历史
- 最近验证摘要与验证入口
- 当前环境 lane 与最近稳定回退边界

读取完成后，本轮执行应只声明一个清晰推进单元，不在同一轮并行吞入多个无关目标。

## 4. 读取产物

执行上下文的读取结果至少应能支持：

- 声明本轮单一推进单元
- 绑定唯一恢复主入口
- 确认当前工作现场与验证入口
- 判断是否仍处于已批准范围内
- 判断本轮结束后是否进入 merge gate

## 5. 边界约束

- 不得只靠 issue 编号、分支名或聊天上下文识别当前事项
- 不得让执行、恢复、review 与 merge gate 分别使用不同的上下文基线
- 不得在执行上下文载体中并行 authored `goal`、`next_step` 或 `blockers`
- 若某字段只是下游仓库局部便利，不得提升为 Loom 固定字段
- 本文件不重复定义执行阶段顺序；阶段顺序以 [execution-chain.md](./execution-chain.md) 为准
